# v1.0.1 retrieval safety hotfix

This hotfix keeps the existing PostgreSQL full-text search and pgvector stack.
It does not introduce a second vector database or a Weaviate service.

## Changes

- Adds a read-only Russian synonym SQLite index as a fallback query-expansion layer.
- Canonicalizes common Russian inflections and conservative Faervell entity spellings.
- Enforces LORE/MECHANICS corpus separation at packet construction.
- Rejects contaminated Discord chunks, concatenated location exports, raw retrieval
  dumps, debuff notifications, and known explicit fragments.
- Reranks lore results by requested intent, entity title, official-wiki provenance,
  and root-article preference.
- Sends only short relevant excerpts to the actor instead of raw chunks.
- Adds deterministic grounded responses for high-confidence ruler queries.
- Adds retrieval diagnostics to logs and registers a knowledge gap when no safe hit
  remains.
- Removes public references to portals, OTN, economic databases, and indexes from
  quest dialogue. Portal quest templates are excluded at runtime.
- Fixes the two mypy class-attribute errors introduced by the v1.0.0 hotfix.
- Adds dry-run/apply quarantine tooling that marks contaminated knowledge chunks
  `GM_ONLY` without deleting the append-only message archive.

## Operational commands

Dry-run contamination report:

```bash
docker cp scripts/quarantine-contaminated-knowledge-v1.0.1.py app-app-1:/tmp/
docker exec app-app-1 python /tmp/quarantine-contaminated-knowledge-v1.0.1.py \
  --report /tmp/knowledge-quarantine.json
```

Apply quarantine:

```bash
docker exec app-app-1 python /tmp/quarantine-contaminated-knowledge-v1.0.1.py \
  --apply --report /tmp/knowledge-quarantine-applied.json
```

Audit source templates:

```bash
python3 scripts/audit-rp-templates-v1.0.1.py
```

After applying quarantine, re-run the official source ingestion if needed. The
runtime safety layer remains active even before reindexing.

## Retrieval logs

Each search writes a structured `knowledge_retrieval` event with the original
query, selected strategy (`ordinary` or `synonym_fallback`), added synonym terms,
safe hit IDs/titles, and rejected hit IDs/titles/reasons. It never logs the full
rejected chunk body.

```bash
docker compose logs --since=10m app | grep knowledge_retrieval
```
